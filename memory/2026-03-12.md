# 2026-03-12 - 周四

## 故障处理器整改
- 今日围绕一次 Gateway 在故障窗口内两次收到 `SIGTERM` 的事件做了系统性排查；当前结论偏向“外部终止/管理动作”而非 Telegram 或 Gateway 自身崩溃。
- 对故障处理链路完成一轮保守化整改：`ghost-process-cleanup.sh`、`network-failover.sh`、`token-monitor.sh`、`fault-handler-controller.sh` 均已重构为以观测、日志、审计为主的保守模式。
- `ghost-process-cleanup.sh` 已改为默认只审计、不自动 kill 主 Gateway；多实例场景写入 `audit-critical.log`，避免误伤。
- `network-failover.sh` 已取消直接调用 Telegram Bot 外发消息与激进恢复动作，仅保留探测、状态落盘与状态切换审计。
- `token-monitor.sh` 已取消自动 fallback 切换与直接告警外发，改为只观测 provider 状态并写入 `~/.openclaw/.provider_status`。
- 新增统一全局开关文件：`~/.openclaw/fault-handlers.env`，当前默认：`SAFE_MODE=1`、`DRY_RUN=0`。
- 已统一故障处理器日志目录到 `~/.openclaw/fault-handlers/`，并补齐 `audit-critical.log`。
- crontab 已验证仍指向故障处理控制器，且 cron 汇总日志路径已改到 `~/.openclaw/fault-handlers/cron.log`。

## 当前判断
- 本轮整改的重点不是增强自动恢复，而是先让 fault handlers 变得可解释、可控、可审计。
- 当前建议继续保持 `SAFE_MODE=1`、`DRY_RUN=0`，先观察 3–5 天，再决定是否需要放开更主动的动作。
