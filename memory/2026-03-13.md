## 根因判定与最小修复规划
- 针对“重复同一条消息后卡住、无法推进、最后无回应”的故障，当前阶段已明确采用“先证据判定、再最小修复”的推进方式，不先盲改系统。
- 当前临时根因排序已形成：**H1 > H3 > H4 > H2**。
  - **H1**：主 Telegram 主会话过重 / compaction 超时 / `using current snapshot` fallback，当前判定为**强成立**。
  - **H3**：lane / queue / retry / fallback 收口异常，当前判定为**中高成立**，更像放大器。
  - **H4**：日报/周报发送主体错位（应由 Assistant bot 发送，却卷入主 assistant 回复链），当前判定为**中等成立**。
  - **H2**：双发送权 / 时区 / 幂等不足，已确认是**真实存在的背景结构问题**，但暂不视为这次“卡住”现象的第一直接主因。
- 当前较稳的一句话结论：这次“重复同一条 → 卡住 → 不推进 → 沉默”的主轴，更像是**主会话过载后进入 timeout + snapshot fallback，再叠加 queue / retry / 收口异常**；双发送权问题更像背景脆弱性而不是首触发因。
- 已固化 4 份工作文档，后续继续排查/执行时应优先沿这条文档链推进：
  - `ROOT_CAUSE_VALIDATION_CHECKLIST_v1.md`
  - `ROOT_CAUSE_MINIMAL_VALIDATION_RUNBOOK_v1.md`
  - `MINIMAL_FIX_PREFLIGHT_CHECKLIST_v1.md`
  - （讨论中已形成但尚未单独落盘为文件名的）`MINIMAL_FIX_PLAN_v1` / “可执行变更清单 v1” 逻辑
- 最小修复方向已被压缩为 3 个核心目标：① 减少主会话承压；② 禁止 timeout / snapshot fallback 后继续正文外发；③ 将日报/周报恢复到 Assistant bot 的正确发送主体。
