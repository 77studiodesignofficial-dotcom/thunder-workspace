# FAULT_HANDLER_OPERATING_MODE

> 目的：定义 Thunder 故障处理器的运行模式、开关含义、默认策略与切换原则。

## 1. 配置文件位置

全局运行模式配置文件：
- `~/.openclaw/fault-handlers.env`

当前字段：
```bash
SAFE_MODE=1
DRY_RUN=0
```

---

## 2. 开关定义

## SAFE_MODE
### 含义
控制故障处理器是否采用保守模式运行。

### 取值
- `1`：保守模式（默认）
- `0`：非保守模式（仅在明确批准后才考虑）

### SAFE_MODE=1 时的行为
- 不自动 kill 主 Gateway
- 不自动切换 fallback
- 不直接调用 Telegram Bot 外发消息
- 以探测、记录、审计为主

### SAFE_MODE=0 时的原则
即使关闭 SAFE_MODE，也**不代表自动允许高风险动作直接执行**。
当前脚本仍按保守实现为主。
如需真正启用激进策略，应先：
1. 明确目标动作
2. 增加 pre_action / result / post_check 审计
3. 单独审批
4. 先在受控条件下验证

---

## DRY_RUN
### 含义
控制脚本是否进入演练模式。

### 取值
- `1`：演练模式
- `0`：正常模式（默认）

### DRY_RUN=1 时的设计目标
- 允许跑探测与日志
- 不执行任何未来可能接入的高风险动作
- 用于验证调度、日志、状态落盘是否正常

### 当前状态
当前脚本即使在 `DRY_RUN=0` 下，也因 `SAFE_MODE=1` 保持保守。
因此：
- `DRY_RUN=1` 更偏向“明确标记为演练”
- `DRY_RUN=0` 更偏向“真实观测模式”

---

## 3. 当前推荐默认值

```bash
SAFE_MODE=1
DRY_RUN=0
```

### 原因
这是当前最适合稳定运行的组合：
- 可以持续观测真实环境
- 可以积累日志与时间线
- 不会因为脚本误判带来额外伤害

---

## 4. 模式切换建议

## 场景 A：日常稳定运行
推荐：
```bash
SAFE_MODE=1
DRY_RUN=0
```

适用：
- 正常生产运行
- 仍在观察故障率
- 尚未完全信任自动恢复逻辑

---

## 场景 B：验证调度与日志
推荐：
```bash
SAFE_MODE=1
DRY_RUN=1
```

适用：
- 新装脚本后首次验证
- 检查 cron 是否正常触发
- 检查日志格式是否正确

---

## 场景 C：计划测试更主动的恢复逻辑
推荐前提：
- 先单独审查脚本
- 先准备审计日志
- 先定义回滚方案

不建议直接将生产环境切到：
```bash
SAFE_MODE=0
DRY_RUN=0
```

除非：
- 风险已被明确接受
- 目标动作非常明确
- 已完成受控测试

---

## 5. 当前各脚本的模式语义

## ghost-process-cleanup.sh
- `SAFE_MODE=1`：只审计，不 kill
- `DRY_RUN=1`：行为同上，但更明确表示演练

## network-failover.sh
- 当前无破坏性动作
- 模式主要用于写入日志字段、保留未来扩展边界

## token-monitor.sh
- 当前无自动切换动作
- 模式主要用于写入日志字段、保留未来扩展边界

## fault-handler-controller.sh
- 统一读取环境文件
- 将模式变量注入到所有子脚本
- `status` 会显示当前模式

---

## 6. 操作建议

### 查看当前模式
```bash
cat ~/.openclaw/fault-handlers.env
```
或：
```bash
~/workspace/fault-handler-controller.sh status
```

### 修改模式后生效方式
这些脚本由 cron 定时触发，修改 `fault-handlers.env` 后，下次运行即会读取新值。

---

## 7. 当前结论

当前故障处理器体系的设计原则是：

> **默认保守，先观测；高风险动作必须后移，不能让脚本自己“想当然”执行。**

在未来没有完成更严格验证前，建议一直保持：
```bash
SAFE_MODE=1
DRY_RUN=0
```
