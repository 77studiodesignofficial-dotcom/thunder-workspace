# INTELLIGENCE_SYSTEM_ROADMAP_v1

> 目标：为 Thunder 构建一套从“长期可用的智能助理”逐步演进到“高自治、强约束、可监督、可回滚的执行系统”的总体路线图。

---

# 1. 总体目标定义

## 1.1 我们要构建的不是普通助手
当前目标不应仅定义为“长期可用的智能助理”，而应定义为：

> **一个在 Boss 定义的目标、边界、权限和监督框架下，能够长期稳定地感知、记忆、判断、拆解、执行、复盘和优化的高自治执行系统。**

这意味着系统不仅要：
- 回答问题
- 执行任务
- 保持连续性

还要逐步具备：
- 目标驱动能力
- 任务调度能力
- 多 agent 分工能力
- 可观测性
- 可恢复性
- 知识检索增强能力

---

## 1.2 边界定义
本路线图追求的是：

> **高自治、强约束、可监督、可回滚**

而不是：
- 无边界自主扩张
- 自行定义长期目标
- 绕开审批与权限体系
- 失控式自动化

因此，自治的前提必须是：
1. 目标来自 Boss 或被 Boss 授权
2. 边界清楚
3. 关键动作可审计
4. 高风险动作可停止、可回退

---

# 2. 当前阶段判断

## 2.1 当前系统所处位置
若把系统成熟度分成 5 个级别：

### Level 0：响应式助手
- 你问，我答
- 无记忆、无连续性

### Level 1：带记忆的执行助理
- 能跨会话保留部分连续性
- 能做多步任务

### Level 2：可调度的主脑
- 能拆任务
- 能调子 agent
- 能做隔离执行和汇总

### Level 3：目标导向自治系统
- 能围绕既定目标持续推进
- 能主动做阶段性管理、复盘、优化

### Level 4：高自治运营系统
- 拥有目标系统、上下文治理、知识检索、任务调度、恢复与评估闭环

### 当前判断
Thunder 当前大致处于：

> **Level 1.5 → Level 2 起步阶段**

理由：
- 已有跨会话记忆骨架
- 已有 cron 与运维机制
- 已具备隔离子会话能力
- 已开始显式建设反馈协议与约束矩阵
- 但尚未形成统一的上下文治理、目标系统、知识系统与调度系统

# 4. 总体架构：12 个模块

为了实现最终目标，建议将系统拆成 12 个模块。

## A. 身份与治理层
### 1. Soul / Persona System
定义“我是谁、怎么协作、怎么判断边界”。

### 2. Goal System
定义 Boss 的长期方向、阶段目标、近期目标与优先级。

### 3. Authority / Approval Model
定义哪些事情可直接做，哪些需先说明，哪些必须审批。

---

## B. 认知与上下文层
### 4. Static Context
长期稳定、不常变化的上下文层。

### 5. Dynamic Context
为当前任务临时调入、阶段性变化的上下文层。

### 6. Memory System
负责短期 / 日记式 / 长期记忆的沉淀、升级与调用。

---

## C. 执行与调度层
### 7. Multi-Agent Operating Model
定义主脑与子 agent 的关系、角色、职责分工。

### 8. Task Orchestration
定义任务拆解、分发、队列、验收、复盘与调度。

### 9. Recovery / Rollback
定义失败止损、回退、恢复、人工接管机制。

---

## D. 知识与检索层
### 10. Knowledge Architecture
定义规则、手册、业务知识、参考资料应放在哪一层。

### 11. RAG Bootstrap
为未来业务知识检索建立最小骨架。

---

## E. 观测与演化层
### 12. Observability / Evaluation
衡量系统是否更稳定、更准确、更省 token、更少失真。

---

# 5. 关键概念定义

## 5.1 Static Context（静态上下文）
### 定义
相对稳定、跨任务长期有效、不应被频繁重写的上下文。

### 典型内容
- Soul
- 用户偏好
- 身份关系
- 长期约束
- 长期记忆摘要
- 默认执行规则

### 原则
- 少而精
- 高稳定
- 高复用

---

## 5.2 Dynamic Context（动态上下文）
### 定义
围绕当前任务、当前阶段、当前问题临时调入的上下文。

### 典型内容
- 最近日志
- 当前任务文档
- 本日记忆
- 本轮结论
- 临时状态文件
- 子 agent 回传结果

### 原则
- 只带当前最相关信息
- 用完即收敛
- 有价值的才升级进长期记忆

---

## 5.3 Memory System（记忆系统）
### 定义
将“会话内容易丢失的信息”转化为“外部可持续的连续性载体”的系统。

### 分层建议
- 工作记忆：当前对话 / 当前任务
- 近期记忆：daily memory
- 长期记忆：`MEMORY.md`
- 程序性记忆：规范、runbook、矩阵文档

### 目标
不是“记录越多越好”，而是：
- 记住重要的
- 召回相关的
- 避免失真和冗余

---

## 5.4 Multi-Agent Operating Model（多 agent 运作模型）
### 定义
由主脑负责理解、拆解、分派、汇总、验收；子 agent 负责在隔离上下文中执行具体工作。

### 目标
- 降低主会话污染
- 提高并行能力
- 保留单一对外交互界面

---

## 5.5 RAG Bootstrap
### 定义
让系统在需要时从知识库中“按需检索”信息，而不是把所有资料都塞进上下文窗口。

### 目标
- 降低上下文膨胀
- 提高知识调用精度
- 为业务接入做准备

---

# 6. 当前核心矛盾

当前系统最大的中期矛盾不是能力不足，而是：

> **上下文治理、记忆治理、任务调度和知识治理还没有上升为统一架构。**

这会带来两个直接问题：

## 6.1 认知失真
- 长上下文稀释重点
- 重要约束被埋没
- 临时结论与长期记忆混杂
- 任务之间相互污染

## 6.2 系统成本上升
- token 消耗增加
- 响应延迟上升
- 调试难度上升
- 会话越长越不稳定

因此，接下来的工作主线应该是：

> **从“会用上下文”走向“治理上下文”**

---

# 7. 未来路线：三阶段推进

## Phase 1：定义层（现在开始）
### 目标
把所有关键概念、边界、模块和相互关系定义清楚。

### 核心任务
1. `CONTEXT_MODEL_v1.md`
2. `MULTI_AGENT_OPERATING_MODEL_v1.md`
3. `MEMORY_SYSTEM_ROADMAP_v1.md`
4. `RAG_BOOTSTRAP_PLAN_v1.md`
5. `SOUL_ARCHITECTURE_v1.md`
6. `GOAL_SYSTEM_v1.md`
7. `AUTHORITY_MODEL_v1.md`

### 产出要求
- 每个概念必须有定义
- 每个模块必须有边界
- 必须明确当前状态、目标状态、过渡路径

---

## Phase 2：机制层（1–4 周）
### 目标
把概念变成可运行机制。

### 核心任务
- 建立静态 / 动态上下文调入规则
- 建立主脑 → 子 agent 的任务模板
- 建立记忆升级 / 去重 / 归档规则
- 建立目标 → 任务 → 子任务的调度规则
- 建立权限与审批的实际操作口径

### 成果
- 主会话更聚焦
- 子任务更隔离
- 记忆更稳定
- token 使用更可控

---

## Phase 3：系统层（中后期）
### 目标
向高自治系统演进。

### 核心任务
- 建立最小 RAG 骨架
- 接入目标驱动任务循环
- 建立复盘与优化机制
- 建立系统评估指标
- 建立业务知识层

### 成果
- 从“做任务”升级为“持续推进目标”
- 从“临时用知识”升级为“知识驱动执行”

- 立刻上重量级 RAG 工程

- 未做规则前提升自治权限

原则：
> **先定义，再机制化，最后系统化。**

# 10. 成功标准

本路线图推进是否有效，不看“文档写了多少”，而看以下 8 个结果：

1. 主会话是否更聚焦
2. 长上下文是否更少失真
3. 重要约束是否更少被淹没
4. 子任务是否更适合隔离执行
5. 记忆是否更稳定、少丢失、少冲突
6. token 消耗是否下降或更可控
7. 响应延迟是否下降或更可控
8. 系统是否更接近目标驱动而不是纯响应驱动

- 运维基础设施
- 规则层雏形

> **能在你的方向与约束下，持续、稳定、可解释地自治推进。**
