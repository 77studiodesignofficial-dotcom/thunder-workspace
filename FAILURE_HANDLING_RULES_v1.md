# FAILURE_HANDLING_RULES_v1

> 目标：为 OpenClaw 当前自动化体系建立统一、可执行、可审计的失败处理规则。

## 1. 适用范围

本规则适用于以下任务：
- 每日本地晨报
- 今日日结
- 周报 / weekly-review
- 健康检查
- 故障处理器（ghost-process-cleanup / network-failover / token-monitor）
- 后续纳入主链路的周期性任务

---

## 2. 失败处理总原则

### 原则 1：失败不能静默吞掉
- 所有主链路失败都必须至少留下日志
- 关键失败必须可追溯到时间、任务、原因

### 原则 2：先分类，再处理
- 不允许所有失败都一律重试
- 必须区分“瞬时失败”与“结构性失败”

### 原则 3：高风险动作宁可保守，不要激进
- 遇到 kill / restart / config 相关动作，优先停止扩散
- 不因一次异常自动触发连锁重启

### 原则 4：主链路优先恢复交付，旁路优先隔离影响
- 主链路关注稳定交付
- 旁路关注不干扰生产链路

---

## 3. 失败分级（v1）

## Level A：瞬时失败（可自动重试）
### 定义
通常由外部波动导致，重复一次可能恢复。

### 典型场景
- 上游模型瞬时错误
- provider 临时不可用
- 短时网络抖动
- 一次性超时

### 处理策略
- 允许自动重试 1 次
- 重试前记录：任务名、失败类型、原始错误摘要
- 若重试成功：记录“已恢复”
- 若重试仍失败：升级为 Level B 或 Level C

### 适用任务
- 晨报
- 日结
- 周报
- 外部 API 查询类轻任务

---

## Level B：稳定性失败（有限恢复，需告警）
### 定义
不是一次性抖动，已经影响交付稳定性。

### 典型场景
- 连续两次同类超时
- 连续两次上游调用失败
- Telegram/通道状态异常持续存在
- 周期任务连续多次未成功送达

### 处理策略
- 停止继续盲目重试
- 记录最近一次成功时间
- 发送简要异常汇报
- 等待人工确认或下一轮有条件恢复

### 适用任务
- 晨报 / 日结 / 周报
- 健康检查告警
- heartbeat 巡检（未来）

---

## Level C：结构性失败（禁止自动重试）
### 定义
根因在本地逻辑、配置或权限层，自动重试几乎无意义。

### 典型场景
- 配置错误
- API key 缺失 / 权限错误
- prompt / 脚本逻辑错误
- 文件路径错误
- 固定参数错误
- provider 不支持某动作

### 处理策略
- 不自动重试
- 记录完整失败上下文
- 标记为“需修复”
- 直接汇报 Boss

### 适用任务
- 所有任务

---

## Level D：高风险失败（立即冻结自动动作）
### 定义
已经涉及主服务可用性、数据安全或误伤风险。

### 典型场景
- Gateway 被异常 SIGTERM
- 故障处理器误杀主进程
- 重复实例/守护冲突
- 配置修改后主服务异常
- kill / restart 逻辑无法证明目标准确

### 处理策略
- 立即停止进一步自动处置
- 禁止继续自动 kill / restart
- 记录 PID、命令行、时间、触发脚本
- 进入人工确认
- 故障期间优先保全主链路

### 适用任务
- ghost-process-cleanup
- network-failover
- update / restart / config 相关任务

---

## 4. 任务级失败策略

## 4.1 每日本地晨报
- 默认超时：60s
- 允许：1 次自动重试
- 连续失败：降级为告警，不继续重试
- 汇报：失败时用一句话说明“上游瞬时错误 / 本地结构错误 / 通道异常”

## 4.2 今日日结
- 默认超时：90s
- 允许：1 次自动重试
- 若连续失败：必须汇报
- 优先保留“最简版日结”能力，避免完全无输出

## 4.3 周报
- 默认超时：120s
- 允许：1 次自动重试
- 若失败：可转人工补跑，不做多次自动重试

## 4.4 健康检查
- 正常静默
- 异常记录
- 关键异常（服务不可达/通道不可用/重复实例）必须告警

## 4.5 ghost-process-cleanup
- 不允许仅凭模糊匹配直接 kill
- 若无法准确识别目标，仅记录，不执行
- 一旦涉及主 Gateway，必须先升级为人工确认

## 4.6 network-failover
- 不允许因一次网络探测失败就重启主服务
- 网络异常先观测、再确认、再处理
- 连续失败才允许进入恢复预案

## 4.7 token-monitor
- 仅负责观测与阈值告警
- 不做高风险自动动作，除非未来有明确批准的策略

## 5. 汇报标准

## 轻度失败（A）
格式：
- 任务名
- 失败摘要
- 是否已自动恢复

示例：
- 今日日结触发上游瞬时错误，已自动重试并恢复。

## 中度失败（B）
格式：
- 任务名
- 连续失败次数
- 当前影响
- 建议下一步

示例：
- 今日日结连续两次失败，当前未产出结果，建议改为人工补跑并观察上游稳定性。

## 重度失败（C / D）
格式：
- 结论
- 根因类别
- 当前风险
- 建议动作

示例：
- 检测到主 Gateway 被外部终止信号打断。当前已恢复，但自动故障处理脚本需暂停激进动作并补审计。

---

## 6. 决策表（简版）

| 失败类型 | 是否自动重试 | 是否汇报 | 是否冻结自动动作 |
|---|---|---|---|
| 上游瞬时错误 | 是，最多 1 次 | 失败后汇报 | 否 |
| 单次超时 | 是，最多 1 次 | 重试后仍失败再汇报 | 否 |
| 连续超时 | 否 | 是 | 否 |
| 配置/权限错误 | 否 | 是 | 否 |
| provider 不支持动作 | 否 | 是 | 否 |
| 模糊匹配 kill 风险 | 否 | 是 | 是 |
| Gateway 异常终止 | 否 | 是 | 是 |
