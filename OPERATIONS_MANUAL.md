# Thunder 运维手册 v1.0

## 一、故障预防规则（强制）

### 1.1 浏览器操作超时（强制）

**所有 browser 操作必须设置 timeoutMs**:
```bash
# ✅ 正确
curl -X POST http://localhost:3000/browser \
  -d '{"action":"open","targetUrl":"...","timeoutMs":15000}'

# ❌ 错误 - 可能导致无限等待
curl -X POST http://localhost:3000/browser \
  -d '{"action":"open","targetUrl":"..."}'
```

**超时时间标准**:
| 操作类型 | 超时时间 | 说明 |
|---------|---------|------|
| 页面打开 | 15,000ms | 一般页面加载 |
| 页面截图 | 10,000ms | 截图操作 |
| 元素点击 | 5,000ms | 简单交互 |
| 文本输入 | 5,000ms | 输入操作 |
| 快照获取 | 10,000ms | 页面分析 |

### 1.2 Exec 操作超时（强制）

**所有 exec 操作必须设置 timeout**:
```bash
# ✅ 正确
exec command="..." timeout=30

# ❌ 错误
exec command="..."
```

**超时时间标准**:
| 操作类型 | 超时时间 | 说明 |
|---------|---------|------|
| 简单命令 | 15秒 | ls, cat, grep 等 |
| 网络请求 | 30秒 | curl, wget 等 |
| 安装操作 | 120秒 | brew install 等 |
| 复杂任务 | 300秒 | 长时任务 |

### 1.3 长时间操作处理

**必须使用 background 模式**:
```bash
# ✅ 正确 - 后台执行，不阻塞
exec command="brew install gh" background=true yieldMs=10000
process action=poll sessionId="xxx" timeout=60000

# ❌ 错误 - 前台执行，可能阻塞数分钟
exec command="brew install gh"
```

## 二、健康检查机制

### 2.1 启动前检查

执行任何任务前，先检查：
```bash
# 检查系统状态
./thunder-ops.sh health

# 检查是否有僵尸进程
./thunder-ops.sh cleanup
```

### 2.2 任务执行中检查

**超过 30 秒的任务，必须先发送进度汇报**:
```
Boss: "执行复杂任务"
Thunder: "收到，此任务预计需要 60 秒，开始执行..."
        ↓（执行中）
Thunder: "进度 50%，预计还需 30 秒..."
        ↓（完成）
Thunder: "任务完成，结果如下..."
```

### 2.3 异常检测

**自动检测指标**:
- 响应时间 > 60 秒 → 报警
- 浏览器操作失败 > 3 次 → 切换备用方案
- 网络请求失败 → 重试 3 次后报错

## 三、故障响应流程

### 3.1 延迟响应处理

**如果 Boss 发现我长时间未响应**:
```
Boss: status 或任意消息
    ↓
Thunder: 立即检查会话状态
    ↓
如果发现有阻塞操作 → 终止并汇报
    ↓
恢复正常响应
```

### 3.2 自动恢复机制

**系统级保护**:
```bash
# 每个会话最多执行 5 分钟
MAX_SESSION_TIME=300

# 每个命令最多执行 60 秒
MAX_COMMAND_TIME=60

# 浏览器操作最多 15 秒
MAX_BROWSER_TIME=15
```

## 四、监控告警

### 4.1 日志记录

**所有操作记录到日志**:
```bash
~/.openclaw/operation.log  # 操作日志
~/.openclaw/health-check.log  # 健康检查日志
~/.openclaw/error.log  # 错误日志
```

### 4.2 告警阈值

| 指标 | 阈值 | 动作 |
|------|------|------|
| 响应时间 | > 30 秒 | 发送进度汇报 |
| 响应时间 | > 60 秒 | 报警并尝试恢复 |
| 错误率 | > 10% | 暂停任务，检查系统 |
| 内存使用 | > 80% | 清理缓存，重启服务 |

## 五、最佳实践清单

### 5.1 任务执行前
- [ ] 评估任务耗时
- [ ] 设置合适的超时时间
- [ ] 准备备用方案
- [ ] 如果 >30 秒，先发送预估时间

### 5.2 任务执行中
- [ ] 定期检查进度
- [ ] 每 30 秒发送一次进度汇报（长任务）
- [ ] 监控资源使用
- [ ] 准备随时终止和恢复

### 5.3 任务执行后
- [ ] 记录执行时间
- [ ] 检查是否有残留进程
- [ ] 更新状态到 MEMORY.md
- [ ] 发送完成汇报

## 六、紧急联系

**如果遇到严重故障**:
1. 发送 "restart" 尝试重启会话
2. 发送新消息重置状态
3. 如果仍无响应，等待系统自动恢复
4. 记录故障时间和现象，事后分析
