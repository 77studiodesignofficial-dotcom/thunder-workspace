# AUTOMATION_TASK_MATRIX_v1

> 目标：梳理当前自动化任务，明确主链路 / 旁路、触发方式、模型、超时、重试与汇报要求。

## 1. 分类原则

### 主链路
满足以下任一条件的任务，归入主链路：
- 定时执行且需要稳定交付
- 与 Boss 日常决策/工作节奏直接相关
- 失败会影响日常协作体验

### 旁路
满足以下任一条件的任务，归入旁路：
- 实验性、探索性、非固定节奏
- 长耗时、成本较高、稳定性不确定
- 可以按需触发，不应挤占主链路资源

## 3. 主链路执行标准（v1）

### 主链路任务统一要求
- 默认使用轻量、稳定、低成本链路
- 必须定义明确超时
- 必须有失败分类
- 必须有最少一层日志
- 失败不能静默吞掉

### 主链路失败处理分级

#### A. 可自动重试
- 上游模型瞬时错误
- 短时网络波动
- 临时 provider 超时

处理：
- 最多自动重试 1 次
- 重试后仍失败则进入告警/人工复核

#### B. 不应自动重试
- 配置错误
- 权限错误
- prompt/逻辑结构错误
- 本地脚本匹配逻辑有误

处理：
- 立即记录
- 停止继续重试
- 汇报 Boss

## 4. 旁路任务执行标准（v1）

### 旁路统一要求
- 不得挤占主链路稳定性
- 优先按需触发
- 优先隔离执行
- 允许更长超时，但必须可中断
- 默认失败不升级为生产事故

### 旁路升级为主链路的条件
若某个旁路任务满足以下条件，可考虑升级为主链路：
- 连续 1 周内被高频重复使用
- 对 Boss 决策/节奏形成固定依赖
- 已有稳定模型/接口/超时策略
- 已定义明确的失败处理与告警方式

## 7. 决策摘要

当前自动化体系应明确采用：
- **主链路轻量化**：晨报 / 日结 / 周报 + 健康与必要故障处理
- **旁路隔离化**：调研、实验、长耗时分析、API 验证
- **失败可解释化**：超时、失败、重试都要可追踪
- **高风险动作审计化**：kill / restart / config 相关动作必须加强记录
