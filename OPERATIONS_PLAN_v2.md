# Thunder 运维机制总体规划 v2.0

## 一、现状问题诊断

### 已部署但未整合的机制

| 机制 | 状态 | 问题 |
|------|------|------|
| 超时控制 | ✅ 已配置 | 分散在各脚本中，无统一标准 |
| 健康检查 | ✅ 已部署 | thunder-ops.sh 使用率低 |
| Cron 任务 | ✅ 运行中 | 3个任务，监控不足 |
| 故障处理器 | ✅ 已部署 | 4个脚本，未整合到 crontab |
| 自动恢复 | ✅ 已部署 | caffeinate 运行中，但无监控 |
| 日志记录 | ⚠️ 分散 | 多个日志文件，无统一查看方式 |

### 核心问题

1. **机制碎片化** - 每个方案独立，无统一调度
2. **监控盲区** - 部署了但不知道是否在工作
3. **报告缺失** - Boss 无法直观了解系统健康状态
4. **故障响应慢** - 发现问题后才处理，非预防性

## 二、运维体系重构方案

### 2.1 三层运维架构

```
┌─────────────────────────────────────────────────────────┐
│ Layer 3: 业务层（Boss 可见）                             │
│ ├── 每日晨报（系统状态摘要）                             │
│ ├── 异常告警（实时推送）                                 │
│ └── 月度报告（汇总分析）                                 │
├─────────────────────────────────────────────────────────┤
│ Layer 2: 控制层（Thunder 自治）                          │
│ ├── 统一监控面板（thunder-dashboard）                    │
│ ├── 故障自动处理（已部署的4个处理器）                    │
│ └── 资源调度（Token/网络/进程）                          │
├─────────────────────────────────────────────────────────┤
│ Layer 1: 执行层（系统级）                                │
│ ├── caffeinate（防休眠）                                 │
│ ├── crontab（定时任务）                                  │
│ └── 系统日志（统一收集）                                 │
└─────────────────────────────────────────────────────────┘
```

### 2.2 职责分工

| 层级 | 负责方 | 职责 | 交付物 |
|------|--------|------|--------|
| **业务层** | Thunder → Boss | 状态汇报、告警通知、建议提供 | 晨报、告警消息、月报 |
| **控制层** | Thunder（自治） | 监控、故障处理、资源管理 | 自动修复、状态记录 |
| **执行层** | 系统/脚本 | 基础设施、定时执行、日志收集 | 后台运行、日志文件 |

### 2.3 关键决策：是否输出运维报告

**建议：是，但精简**

理由：
- Boss 需要了解系统运行状况
- 但不需要技术细节
- 重点：健康状态 + 待处理问题 + 建议

**报告体系设计**：

```
实时层（自动）
├── 故障告警 → Telegram 即时推送
└── 恢复通知 → Telegram 即时推送

日度层（Cron 任务）
├── 晨报：系统状态摘要（已部署）
└── 日结：今日异常汇总（可扩展）

周度层（人工触发）
└── 周报：运行数据分析 + 优化建议

月度层（人工触发）
└── 月报：完整运维总结 + 规划建议
```

## 三、统一监控面板（核心）

### 3.1 thunder-dashboard 设计

整合所有监控到一个入口：

```bash
# 使用方式
./thunder-dashboard [command]

Commands:
  status      # 查看整体健康状态（给 Boss 看）
  health      # 详细健康检查（技术诊断）
  logs        # 查看最近日志
  report      # 生成运维报告
  fix         # 执行一键修复
```

### 3.2 status 输出示例（给 Boss 看）

```
⚡ Thunder 系统状态 - 2026-03-08 09:00

🟢 整体健康度: 95/100

核心服务:
├── 🟢 OpenClaw Gateway: 运行中 (12小时)
├── 🟢 Telegram Bot: 在线
├── 🟢 Moonshot API: 正常 (余额充足)
├── 🟢 智谱 GLM: 正常 (免费额度)
└── 🟢 GitHub CLI: 已认证

今日异常:
├── ⚠️ 凌晨 02:07 系统休眠（已恢复）
└── ✅ 无其他异常

待处理事项:
├── 🔴 配置 Brave Search API Key
└── 🟡 提交工作区文件至 Git

建议:
└── 系统休眠问题已解决（caffeinate 运行中）
```

### 3.3 技术实现

```bash
#!/bin/bash
# thunder-dashboard.sh

case "$1" in
    status)
        # 生成给 Boss 看的摘要
        generate_boss_report
        ;;
    health)
        # 技术详细诊断
        run_health_checks
        ;;
    logs)
        # 聚合所有日志
        show_recent_logs
        ;;
    report)
        # 生成完整报告
        generate_full_report
        ;;
    fix)
        # 一键修复常见问题
        auto_fix_issues
        ;;
esac
```

## 四、运维报告体系（精简版）

### 4.1 实时告警（自动）

**触发条件**:
- 服务崩溃 → 立即告警
- Token 耗尽 → 立即告警 + 自动切换
- 网络掉线 > 1分钟 → 告警
- 幽灵进程 > 1个 → 告警 + 自动清理

**输出**: Telegram 消息

### 4.2 每日晨报（已部署，需优化）

**当前问题**:
- 资讯收集依赖 web_search（缺少 API Key 失败）
- 内容过长

**优化后**:
```
📅 Thunder 日报 - 3月8日

健康度: 🟢 95/100
├── Gateway: 运行 12h
├── APIs: 全部正常
└── 故障: 1次（已恢复）

待办: 2项
├── 🔴 Brave API Key
└── 🟡 Git 提交

一句话总结: 系统稳定，建议配置搜索 API
```

### 4.3 周度报告（建议新增）

**触发**: 每周日晚，或 Boss 发送 `/weekly-report`

**内容**:
```
📊 Thunder 周报 - 第10周

运行统计:
├── 在线时长: 167/168 小时 (99.4%)
├── 处理任务: 47 个
├── API 调用: 1,234 次
└── 故障次数: 2 次（均自动恢复）

资源消耗:
├── Moonshot: ¥12.5
├── 智谱 GLM: ¥0 (免费)
└── 网络流量: 正常

优化建议:
└── 配置 Brave API Key 可提升资讯质量
```

### 4.4 月度报告（建议新增）

**触发**: 每月1日，或 Boss 发送 `/monthly-report`

**内容**: 完整运维数据分析 + 长期规划建议
