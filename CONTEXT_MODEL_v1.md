# CONTEXT_MODEL_v1

> 目标：为 Thunder 建立一套可治理的上下文模型，解决长上下文导致的记忆失真、token 消耗、响应延迟、任务污染与知识混杂问题。

---

# 1. 为什么要建立上下文模型

当前系统已经具备：
- 长期记忆骨架
- daily memory
- 子 agent 能力
- 规则文档与运维文档

但如果没有正式的上下文模型，系统会持续面临以下问题：

## 1.1 认知问题
- 长上下文稀释重点
- 重要约束被埋没
- 任务之间互相污染
- 临时结论和长期知识混在一起
- 记忆召回不稳定，出现“记不准”或“记错层”

## 1.2 系统问题
- token 消耗越来越高
- 响应延迟越来越明显
- 长任务更容易超时
- 调试与回溯越来越困难

因此，本模型的核心目标是：

> **让系统不是“堆上下文”，而是“治理上下文”。**

---

# 2. 总体原则

## 原则 1：常驻上下文要少而稳
不是所有重要信息都适合常驻。常驻层只保留长期稳定、高复用的信息。

## 原则 2：临时上下文要短而准
当前任务只带当前最相关的信息，而不是把所有历史都拖进来。

## 原则 3：记忆不等于上下文
记忆是外部存储；上下文是当前调用时实际带入的材料。二者必须区分。

## 原则 4：知识不等于记忆
业务知识、规范、资料、参考手册不应长期混在记忆层里，未来应转入知识层 / RAG 层。

## 原则 5：复杂任务优先隔离
当任务会明显膨胀主会话上下文时，应优先交给子 agent / 隔离会话处理，再回传压缩结果。

---

# 3. 上下文五层模型

建议将 Thunder 的上下文体系分为五层。

## Layer 0：身份与治理层（Identity & Governance）
### 定义
决定系统“是谁、怎么协作、边界在哪”的最顶层上下文。

### 典型内容
- `SOUL.md`
- `USER.md`
- `IDENTITY.md`
- 核心协作规则
- 高优先级边界与审批原则

### 特征
- 高稳定
- 高频复用
- 不应被任务性内容污染

### 作用
确保系统不会因为单次任务而“忘了自己是谁、该怎么做”。

---

## Layer 1：长期记忆层（Long-term Memory）
### 定义
记录对长期协作有价值、跨会话应保留的事实与结论。

### 典型内容
- `MEMORY.md`
- 长期偏好
- 关键决策
- 长期有效的工作方式
- 已完成的关键基础设施结论

### 特征
- 相对稳定
- 周期性维护
- 应去重、去噪、去流水账

### 作用
保证系统跨会话连续性。

---

## Layer 2：近期工作记忆层（Working / Recent Memory）
### 定义
记录近期发生、可能短期内反复引用，但未必值得永久保留的内容。

### 典型内容
- `memory/YYYY-MM-DD.md`
- 最近 1–3 天的重要工作进展
- 当前观察期状态
- 临时异常样本
- 当日决策与阶段结论

### 特征
- 半稳定
- 时间敏感
- 适合阶段性回顾后上收或淘汰

### 作用
让系统不必依赖超长会话来记住“最近刚发生什么”。

---

## Layer 3：任务动态上下文层（Task Dynamic Context）
### 定义
围绕当前任务临时调入的具体材料。

### 典型内容
- 当前任务相关日志
- 当前脚本文件
- 当前文档
- 当前错误信息
- 当前决策分支
- 子 agent 回传结果

### 特征
- 强相关
- 强时效
- 用完即收敛

### 作用
让当前轮只关注“现在要解决什么”。

---

## Layer 4：外部知识与检索层（Knowledge / RAG）
### 定义
存放不适合常驻上下文、但在具体任务中可能被检索调用的知识资料。

### 典型内容
- 技术资料
- 业务资料
- 操作手册
- 规范库
- 历史方案档案
- 外部研究材料

### 特征
- 大体量
- 低常驻性
- 检索优先于常驻

### 作用
避免把大量知识硬塞进主上下文窗口。

---

# 4. 静态上下文 vs 动态上下文

## 4.1 静态上下文定义
静态上下文 = 在较长时间范围内保持稳定、应高频复用的信息集合。

### 包括
- Layer 0：身份与治理层
- Layer 1：长期记忆层中的高稳定部分

### 不应包括
- 当日临时结论
- 当前错误日志
- 某次排障过程的细节
- 大段业务资料

---

## 4.2 动态上下文定义
动态上下文 = 为当前任务或当前阶段临时调入的信息集合。

### 包括
- Layer 2：近期工作记忆的相关片段
- Layer 3：当前任务材料
- Layer 4：被检索出来的知识片段

### 原则
- 按需调入
- 只调最相关部分
- 结束后要么丢弃，要么沉淀

---

# 5. 什么问题该放在哪一层

## 应留在静态层的
- 角色关系
- 长期偏好
- 高优先级规则
- 协作协议
- 长期有效的关键决策

## 应留在近期工作记忆层的
- 本周正在推进的议题
- 当前观察期结论
- 临时异常样本
- 阶段性整改记录

## 应只留在任务动态层的
- 当前日志窗口
- 当前命令输出
- 当前一次性错误上下文
- 当前具体排障细节

## 应进入知识层 / RAG 层的
- 大体量参考文档
- 业务资料
- 技术文档库
- 操作手册
- 历史项目资料

---

# 6. 动态上下文的来源

动态上下文主要有四个来源：

## 6.1 最近记忆召回
来自：
- `memory/YYYY-MM-DD.md`
- 必要时的 `MEMORY.md`

### 用途
回答“最近做过什么”“延续哪个议题”。

---

## 6.2 当前任务材料
来自：
- 当前文件
- 当前日志
- 当前错误
- 当前命令结果

### 用途
解决当前问题本身。

---

## 6.3 子 agent 回传
来自：
- 隔离子任务的结果摘要
- 并行执行后的结构化结论

### 用途
减少主会话直接承载大量探索过程。

---

## 6.4 外部知识检索
来自：
- 本地知识库
- 文档
- 未来的 RAG 检索层

### 用途
按需补知识，而不是硬塞进主会话。

---

# 7. 上下文收敛机制

为了防止上下文无限膨胀，必须建立“收敛”机制。

## 7.1 三种收敛结果
### A. 丢弃
如果内容：
- 一次性
- 无长期价值
- 已完成且不会再引用

则不保留到长期层。

### B. 进入近期记忆
如果内容：
- 接下来几天还会用到
- 属于当前阶段的重要进展
- 需要后续观察

则写入 `memory/YYYY-MM-DD.md`。

### C. 升级到长期记忆 / 规则 / 知识
如果内容：
- 会跨周、跨月继续有效
- 是关键决策或稳定规则
- 是可复用经验

则升级到：
- `MEMORY.md`
- 规范文档
- 知识层文档

---

# 8. 多 agent 在上下文模型中的角色

多 agent 不只是并行工具，而是：

> **主会话减上下文负担的核心机制之一。**

## 8.1 主脑负责什么
- 任务理解
- 上下文选取
- 是否拆分子任务
- 结果汇总
- 最终判断

## 8.2 子 agent 负责什么
- 在隔离上下文中探索
- 处理局部复杂任务
- 返回压缩后的结果

## 8.3 适合拆给子 agent 的任务
- 长日志分析
- 大量文件探索
- 深度研究
- 代码实现
- 并行对比分析

## 8.4 不必拆的任务
- 简单问答
- 短判断
- 轻量状态确认

---

# 9. 记忆系统在上下文模型中的角色

记忆系统不是“把所有东西存起来”，而是：

> **为上下文系统提供稳定、低成本、可召回的连续性支持。**

## 记忆系统要解决的不是
- 替代所有上下文
- 保存所有过程细节

## 记忆系统要解决的是
- 让重要结论不依赖长会话存在
- 让近期工作能被低成本召回
- 让长期协作风格不丢失

---

# 10. RAG 在上下文模型中的角色

RAG 不应承担：
- Soul
- 身份关系
- 高优先级边界
- 高稳定长期偏好

RAG 应承担：
- 大体量知识材料的按需检索
- 业务知识的任务期调用
- 规范文档的精准引用

## 简单说
- Soul / Rule / Identity → 静态层
- Memory → 连续性层
- RAG → 知识检索层

---

# 11. 当前系统的阶段判断

## 已经具备
- 静态上下文基础
- 长期 / 近期记忆基础
- 子 agent 隔离能力
- 文档化规则建设起点

## 尚未完成
- 正式的上下文装载规则
- 动态上下文的标准化调入逻辑
- 记忆升级规则的制度化
- RAG 的最小知识层骨架
- 主脑与子 agent 的标准上下文契约

### 结论
当前上下文模型处于：

> **基础存在，但尚未制度化；需要从“经验使用”升级到“显式治理”。**

---

# 12. 现阶段的执行建议

## 立刻应做
1. 明确静态层清单
2. 明确动态层来源清单
3. 明确哪些信息升级进 daily / MEMORY / 规范 / 知识层
4. 明确哪些任务必须优先走子 agent

## 暂时不要直接做
- 把所有历史资料都塞进上下文
- 在 RAG 没分层前就大量建库
- 在没有上下文规则前激进并行多 agent

---

# 13. 成功标准

这个上下文模型是否有效，要看以下结果：

1. 主会话是否更聚焦
2. 重要信息是否更少被淹没
3. 记忆失真是否减少
4. 子任务是否更少污染主会话
5. token 使用是否更可控
6. 响应延迟是否更可控
7. 规则、记忆、知识是否开始分层清晰

# 15. 总结

Thunder 的上下文治理不应再建立在“会话越长越连续”的假设上，而应建立在：

- **静态层稳定**
- **动态层按需调入**
- **记忆层负责连续性**
- **子 agent 负责隔离复杂度**
- **RAG 负责检索知识**

只有这样，系统才能在保持连续性的同时，避免：
- 记忆失真
- token 膨胀
- 响应变慢
- 任务互相污染

这就是从“使用上下文”走向“治理上下文”的核心。
